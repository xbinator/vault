# BEditor 聚焦触发意外回写问题复盘

## 背景

在富文本编辑器中打开 `docs/superpowers/specs/2026-04-26-sessionhistory-virtual-scroll-design.md` 后，即使用户没有修改正文，只要编辑器完成挂载或获得焦点，外部就会收到一次 `update:value`。这会让文档被误判为已变更，进一步带来脏状态、自动保存或 diff 噪音。

本次问题最初表现为“聚焦后文档内容被改写”，实际排查后确认是 Markdown 导入导出链路在无用户编辑的情况下没有保持稳定。

## 问题现象

### 用户可见表现

- 打开指定设计文档后，编辑器可能立即触发一次内容回写。
- 仅执行聚焦操作，不输入任何字符，也会出现内容变更。
- 变更主要体现在文档尾部和表格附近，表现为多出空行，或表格格式被重新序列化。

### 典型复现文件

- `docs/superpowers/specs/2026-04-26-sessionhistory-virtual-scroll-design.md`

### 影响

- 破坏“未编辑即无变更”的基本编辑器预期。
- 增加文档 diff 噪音，影响问题定位与代码评审。
- 可能误触发保存、同步、脏状态提示等上层逻辑。

## 根因分析

这次问题不是单点缺陷，而是两个机制叠加后的结果。

### 1. 表格 Markdown 默认序列化不保留原文

Tiptap 默认的表格 Markdown 渲染会按自己的规则重新生成表格内容，例如：

- 重新计算分隔行样式
- 对单元格内容做规范化
- 不保证与导入时的原始 Markdown 文本完全一致

这意味着即使表格语义没有变化，重新导出后字符串也可能发生变化。

### 2. 文档尾部的 trailing node 机制会在表格后补空段落

`StarterKit` 默认启用 `trailingNode` 插件。该插件会在某些块节点后自动补一个尾随段落，以保证文档末尾始终有一个可编辑块。

在本次场景中：

- 文档最后一个有效块是 `table`
- `trailingNode` 在表格后插入了一个空 `paragraph`
- 该空段落在导出 Markdown 时变成了额外的 `\n\n`

最终效果就是：即使用户什么都没做，导出结果也比原文多出两个换行，从而触发一次无意义回写。

## 解决思路

修复目标不是“禁止所有规范化”，而是确保“未改动内容保持原样导出，真正改动后再走稳定序列化”。

### 方案一：为表格增加原文保留能力

在表格导入阶段记录两类信息：

- `markdownRaw`：原始表格 Markdown 文本
- `markdownSignature`：根据表格结构计算出的签名

在导出阶段：

- 如果当前表格签名与导入时一致，直接回写 `markdownRaw`
- 如果表格结构已变化，再使用受控的 fallback 渲染逻辑生成稳定 Markdown

这样可以避免“仅导入导出一次就重写表格格式”的问题。

### 方案二：禁止在表格结尾自动插入尾随空段落

保留 `trailingNode` 机制的整体能力，但对表格场景做定向排除：

- 配置 `StarterKit.trailingNode.notAfter = ['table']`

这样可以避免文档尾部是表格时被自动补一个空 paragraph，进而消除尾部多出的 `\n\n`。

## 实际改动

### `src/components/BEditor/hooks/useExtensions.ts`

本次修改集中在扩展配置层，主要包括：

- 扩展 `MarkdownTable`，增加 `markdownRaw` 和 `markdownSignature` 属性
- 新增表格签名构建逻辑，按“单元格类型 + 对齐 + 归一化文本”判断内容是否实际变化
- 新增受控的表格 fallback 渲染，保证表格在发生编辑后仍能稳定导出
- 调整 `StarterKit` 的 `trailingNode` 配置，避免在 `table` 后自动补空段落

### 测试

新增并收敛真实文档诊断测试，覆盖两个核心断言：

1. 指定问题文档经过一次富文本导入导出后，字符串必须保持不变
2. 指定问题文档在挂载和聚焦后，不应触发任何 `update:value`

## 为什么这样修

相比“一刀切关闭 trailingNode”或“完全接受表格规范化输出”，当前方案更符合编辑器实际需求：

- 保留编辑器已有能力，只对问题路径做最小收敛
- 未修改内容保持原样，减少无意义 diff
- 真正编辑过表格时，仍可得到稳定、可预测的 Markdown 输出
- 风险集中在表格扩展和文档末尾策略，影响面相对可控

## 验证方式

### 自动化验证

已通过以下回归验证：

- 真实问题文档 round-trip 不变
- 真实问题文档 mount / focus 不触发 `update:value`
- 之前已修复的 HTML 注释场景保持正常
- heading id 补齐事务不改写 Markdown

### 手动验证建议

- 打开 `docs/superpowers/specs/2026-04-26-sessionhistory-virtual-scroll-design.md`
- 不输入任何字符，仅聚焦编辑器
- 确认外层未收到内容变更
- 再对表格内容做一次真实编辑，确认仍能正常保存并导出

## 风险与后续建议

### 当前风险

- 表格签名目前基于“结构 + 归一化纯文本”，如果后续要严格保留单元格内部复杂 Markdown 细节，可能还需要更细粒度签名。
- 这次只对 `table` 结尾做了 `trailingNode` 排除，其他块级节点如果未来出现类似现象，需要按真实场景再补充约束。

### 后续建议

- 增加“文档以表格结尾”的通用回归测试，避免同类问题再次出现。
- 补充 HTML 注释的多行和带空白边界用例，继续收紧 Markdown round-trip 稳定性。
- 如果后续发现更多“未编辑即回写”的案例，可将“原文保留 + 签名判断”的策略扩展到其他高风险块节点。
