# 压缩消息视觉设计

## 背景

当前 `src/components/BChatSidebar/components/MessageBubble/BubblePartCompression.vue` 会直接向用户展示 `summaryText`。这会带来两个问题：

- 压缩消息看起来像一条普通聊天正文，而不是“上下文边界”
- 长摘要会抢占聊天流视觉焦点，影响对真实对话内容的浏览

本次设计目标是让压缩消息只表达“这里发生了一次上下文整理”，不向用户暴露内部摘要原文。

## 目标

- 将压缩消息塑造成时间线式的上下文边界节点
- 不向用户展示 `summaryText`
- 保留成功、失败、进行中三种状态的可识别性
- 让用户明确知道“后续回复将从这里继续”

## 非目标

- 不调整压缩逻辑、摘要生成逻辑和上下文边界判断逻辑
- 不新增摘要展开查看能力
- 不向用户暴露内部结构化摘要原文

## 视觉方向

压缩消息改为“时间线分隔符 + 状态胶囊”的形态，不再使用普通聊天正文块。

- 整体居中显示
- 中间为状态 capsule
- 左右为延伸细线，表达“上下文从这里切换”
- 下方只显示一行简短说明
- 不显示 `summaryText`

## 状态设计

### 成功态

- 主标题：`上下文已压缩`
- 副标题：`此前对话已整理，后续回复将从这里继续`
- 颜色：中性底色，搭配轻量绿色或青色强调

### 进行中态

- 主标题：`正在压缩上下文`
- 副标题：`正在整理此前对话，请稍候`
- 颜色：中性色
- 可附加轻微 loading 动效

### 失败态

- 主标题：`压缩失败`
- 副标题：`未能完成上下文整理，可稍后重试`
- 可选追加一行短错误信息，但不展示技术细节堆栈
- 颜色：低饱和橙红色

## 信息层级

默认只展示以下内容：

- 状态标题
- 一行状态说明
- 可选元信息，如创建时间

以下内容不展示：

- `summaryText`
- 长段落摘要
- 模型生成的内部整理文本

## 组件结构建议

`BubblePartCompression.vue` 建议拆分为以下结构：

- `compression-node`
  - `compression-node__line`
  - `compression-node__pill`
  - `compression-node__line`
- `compression-node__meta`
  - `compression-node__title`
  - `compression-node__description`
  - `compression-node__error`（仅失败态可见）

## 与现有消息流的关系

- 压缩消息仍然保留在消息列表中
- 视觉上应显著区别于 `user` / `assistant` 普通气泡
- 它更像“系统边界节点”，而不是“系统回复”

## 验收标准

- 成功态不再显示 `summaryText`
- 失败态不再显示长摘要内容
- 进行中态能被明显识别
- 在会话流中一眼能看出这是一次“上下文切换”
- 普通聊天消息的阅读节奏不被压缩消息打断
