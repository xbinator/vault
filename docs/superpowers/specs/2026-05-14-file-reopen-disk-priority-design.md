# 2026-05-14 File Reopen Disk Priority Design

## Background

当前 Tibis 对“最近文件”的语义是草稿恢复优先。

- 最近文件记录会持久化 `content` 与 `savedContent`
- 通过欢迎页、最近文件搜索、原生打开入口再次打开文件时，若命中已有记录，当前逻辑会直接复用缓存内容
- 编辑器加载后再通过 `reconcileStoredFileWithDisk()` 判断是否需要和磁盘状态协调

这会让“重新打开文件”更像“恢复上次草稿”，而不是“重新从磁盘读取文件”。

本次调整的目标，是只修改“从外部重新打开文件”的语义，不改变已经打开的编辑会话和顶部标签切换行为。

## Goal

将欢迎页、最近文件、原生打开等入口统一定义为“重新打开文件”入口，并满足以下规则：

1. 如果目标文件当前已经有打开中的标签页，直接回到该标签，不触发磁盘重载。
2. 如果目标文件当前没有打开中的标签页，再次打开时直接读取磁盘最新内容。
3. 对有磁盘路径的文件，不再通过最近文件缓存恢复未保存草稿。
4. 对没有磁盘路径的未保存文档，继续保留当前草稿恢复逻辑。

## Non-Goals

- 不修改顶部 `HeaderTabs` 的切换、关闭、拖拽和激活逻辑
- 不改变当前编辑器内存会话的保存、脏状态和自动保存机制
- 不调整“关闭标签页时的未保存提醒”交互
- 不在本次改动中重新设计最近文件数据模型

## User-Facing Behavior

### Reopen From External Entry

以下入口统一视为“外部重新打开”：

- 欢迎页最近文件
- 最近文件搜索
- 原生“打开文件”

行为规则如下：

- 若文件路径对应的标签页已经打开，直接激活已命中的首个标签页。
- 若文件路径未处于打开状态，读取磁盘内容并覆盖最近文件记录中的 `content` 与 `savedContent`。
- 随后进入编辑器页面，编辑器看到的初始内容就是最新磁盘内容。

### Existing Open Session

如果同一路径文件已经在标签栏中打开：

- 不创建新的最近文件记录
- 不覆盖该标签页当前内存中的未保存内容
- 不触发磁盘与草稿冲突确认

用户会感知为“还是原来的会话，只是切回去了”。

补充约束：

- 当前标签模型没有 `loading` / `error` 状态字段，因此本次设计不引入额外状态判断。
- “已打开”仅定义为标签栏中存在可匹配目标路径的标签。
- 若历史或异常情况下存在多个相同路径标签，外部重新打开时复用首个命中标签，不再额外创建重复标签。

### Unsaved Draft Files Without Path

对于 `path === null` 的未保存文档：

- 仍然按现有逻辑通过 `id` 恢复缓存内容
- 因为没有磁盘真值，不参与“磁盘优先重开”规则

## Design

### Entry-Level Decision

把“是否强制按磁盘重开”的判断放在打开入口层，而不是编辑器会话层。

原因：

- 需求针对的是“从哪里打开”，不是“编辑器如何加载”
- 可以避免把入口语义耦合到 `useSession` 的通用初始化流程
- 顶部标签切换保持原状，不需要传递额外来源状态

### Tabs Check First

在通过路径重新打开文件前，先检查是否已经存在同路径的打开标签。

若存在：

- 直接跳转到该标签的路由
- 终止后续磁盘读取和缓存覆盖

这保证“开着的文件还是原来的会话”。

路径匹配以目标文件绝对路径为准，而不是以最近文件 `id` 为准。

### `openFileById()` Call Flow

`openFileById()` 的新调用链需要显式区分“无路径草稿”和“有路径磁盘文件”：

1. 先按 `id` 读取最近文件记录。
2. 若记录不存在，返回 `null`。
3. 若记录 `path === null`，继续沿用当前 `openExistingFile(id)` 逻辑恢复草稿。
4. 若记录存在 `path`，先检查该路径是否已有打开标签。
5. 若已有打开标签，直接导航到该标签。
6. 若没有打开标签，调用“按路径强制从磁盘刷新最近文件记录”的 store 方法，再导航到编辑器。

这样可以保证 `openFileById()` 仍然是最近文件入口的统一收口点，但内部行为不再等价于“直接恢复缓存正文”。

### Force Refresh Stored File From Disk

若同路径文件当前没有打开标签，则执行“按路径从磁盘强制刷新”：

1. 读取磁盘文件内容
2. 若最近文件中已存在该路径记录，则保留其 `id`、`createdAt` 等身份字段
3. 用磁盘内容覆盖该记录的 `content` 与 `savedContent`
4. 更新 `name`、`ext`、`openedAt`、`savedAt`
5. 若不存在该路径记录，则按现有创建逻辑新增一条记录

这样可以避免因重建记录导致文件 `id` 变化，进而减少对其它依赖文件 ID 的模块造成副作用。

这里的语义前提是：

- 对于有磁盘路径的文件，最近文件 `id` 表示“该路径对应的本地文件记录”，而不是“某一版正文内容”的不可变身份。
- 即使文件被外部程序替换了正文，本次设计仍保留原有 `id`，把它视为同一路径文件的连续记录。
- 如果未来需要区分“同路径但语义已被替换成另一份文档”的场景，应作为单独功能引入额外判定，而不是在本次重开语义调整中顺带处理。

### Concurrency Guard

外部重新打开同一路径文件时，仍需要沿用路径级并发保护，避免快速连续点击触发重复磁盘读取与存储覆盖。

设计要求：

- 同一路径的“强制磁盘刷新”操作应串行化。
- 若已有同路径打开流程在进行中，后续重复触发应复用现有保护策略，避免并发写入最近文件记录。
- 不额外引入跨路径的全局锁，避免影响不同文件并行打开。

## Affected Modules

### `src/hooks/useOpenFile.ts`

- 统一收口“外部重新打开”行为
- 新增“先查是否已有打开标签，再决定是否强制按磁盘打开”的分支
- `openFileById()` 在命中有路径文件时，不再直接调用 `openExistingFile()`

### `src/stores/files.ts`

- 保留 `openExistingFile()` 作为纯 `touch` 逻辑，供现有会话使用
- 新增一个按路径强制读取磁盘并刷新最近文件记录的方法
- 该方法只负责最近文件存储层与磁盘同步，不负责路由

### `src/layouts/default/hooks/useFileActive.ts`

- 顶部文件菜单、快捷键事件与 `file:openRecent` 事件都通过 `useOpenFile` 收口，因此会直接继承新的外部重新打开行为

### `src/views/welcome/index.vue`

- 欢迎页最近文件与“打开文件”入口已经复用 `useOpenFile`，因此会直接继承新逻辑
- 组件本身不需要承载额外状态，只复用打开入口封装

## Error Handling

- 磁盘读取失败时，返回 `null` 或保持现有错误冒泡策略，不写入损坏的最近文件记录
- 如果最近文件记录存在但磁盘文件已不可读，不恢复旧缓存草稿；由调用方决定是否提示或静默失败
- 已打开标签命中时不触发磁盘读取，因此也不会因磁盘异常破坏现有编辑会话
- 当前设计不因外部正文替换而重置最近文件 `id`；路径仍然是有盘文件的主身份锚点

## Testing

需要补充的验证重点：

1. 同路径文件已打开时，从欢迎页/最近文件/原生打开再次触发，应回到已有标签并保留未保存内容。
2. 同路径文件未打开时，从最近文件重新打开，应读取磁盘最新内容，不恢复旧草稿。
3. 没有路径的未保存文档，仍可通过现有逻辑恢复。
4. 首次通过路径打开文件时，仍会创建最近文件记录。
5. 磁盘文件被删除或不可读时，应保持当前打开失败策略，不加载旧缓存正文，也不写入损坏的最近文件记录。
6. 若历史状态中存在多个相同路径标签，外部重新打开时应复用首个命中标签，不继续放大重复标签数量。
7. 快速连续触发同一路径外部重新打开时，不应出现重复磁盘覆盖或竞争性写入。

## Open Question Resolution

拖拽打开文件不在本次范围内，继续保持现有行为。
