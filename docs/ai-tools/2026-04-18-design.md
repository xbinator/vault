# AI Tools 设计记录

日期：2026-04-18

## 背景

当前 `BChat` 已经支持普通多轮聊天和流式回复，消息通过 `useChat` 调用 Electron 主进程的 AI 服务。主进程使用 Vercel AI SDK 的 `streamText` 发起模型请求，并通过 IPC 将 `text-delta`、`finish`、`complete`、`error` 事件传回渲染进程。

现有链路还没有 tool calling 能力：

- `AIRequestOptions` 只包含 `prompt`、`system`、`temperature`、`messages` 等基础字段。
- Electron IPC 只传回文本流和完成事件，没有透传工具调用事件。
- `BChatSidebar` 可以访问当前聊天会话，但还没有把当前编辑器上下文交给 AI 工具层。
- 当前文档内容在 `views/editor` 的 `fileState.content` 中，编辑器实例通过 `BEditorPublicInstance` 暴露了撤销、搜索、聚焦等能力，但还没有明确的插入、替换选区、替换全文等公共编辑动作。

## 目标

第一版实现 AI 内置工具能力，让 AI 可以围绕当前编辑器文档完成读写协作。

MVP 选择“读写编辑工具”：

- AI 可以读取当前文档、当前选区、搜索当前文档。
- AI 可以请求插入内容、替换选区、替换全文。
- 写操作必须经过用户确认后执行。

同时，设计需要为未来扩展保留空间：

- 用户自定义工具。
- MCP/插件式工具。
- 将工具执行器从渲染进程迁移到 Electron 主进程。

## 选定方案

采用“方案 1 + 抽象预留方案 2”：

- 第一版由渲染进程编排工具调用。
- Electron 主进程继续负责模型调用和流式事件。
- 主进程识别并透传模型 tool call 请求，但不直接执行内置编辑工具。
- 渲染进程收到 tool call 后，根据工具权限自动执行只读工具，或弹出确认后执行写工具。
- 工具定义、工具调用、工具结果和执行器都用独立类型描述，不绑定在 `BChat` 内部。

这个方案适合当前项目，因为当前文档状态、选区状态和编辑器实例都在渲染进程。第一版如果让主进程执行工具，会让“读取当前选区”和“替换编辑器内容”变成 IPC 往返，复杂度更高。

但类型边界不能写死在渲染进程。后续如果工具变成文件系统、Shell、MCP 服务或跨窗口能力，可以把执行器迁移到 Electron 主进程，上层聊天 UI 仍然通过同一套工具协议工作。

## 我的判断

这个方向是稳的：

- 当前上下文在 renderer，第一版 renderer 编排最自然。
- 项目已经使用 Vercel AI SDK，主进程可以基于现有 `streamText` 扩展 tools，不需要重写 provider 层。
- 通过 `AIToolDefinition`、`AIToolCall`、`AIToolResult`、`AIToolExecutor` 等边界，可以先快后稳，不堵死未来 custom tools 和 MCP。

需要避免的点：

- 不要让模型绕过确认直接修改文档。
- 不要把内置工具逻辑写死进 `BChat`，否则后续扩展会困难。
- 不要第一版就把所有执行器放到主进程，否则编辑器上下文访问会变复杂。
- 不要用 prompt 伪 JSON 指令代替原生 tools，除非某些 provider 暂时不支持 tools 时作为降级策略。

## 第一版内置工具

### 只读工具

`read_current_document`

- 读取当前文档标题、路径、Markdown 内容。
- 自动执行。
- 适合回答“总结这篇文章”“帮我分析当前文档结构”。

`get_current_selection`

- 读取当前选区文本和范围。
- 自动执行。
- 如果没有选区，返回空选区状态，而不是抛错。

`search_current_document`

- 输入关键词。
- 返回匹配数量和若干上下文片段。
- 自动执行。

### 写入工具

`insert_at_cursor`

- 在当前光标处插入 Markdown。
- 需要用户确认。
- 如果当前没有明确光标，默认聚焦编辑器后插入到当前位置；无法确定时返回失败。

`replace_selection`

- 替换当前选区。
- 需要用户确认。
- 如果没有选区，返回失败，并让模型改用插入或说明需要用户先选择文本。

`replace_document`

- 替换全文。
- 属于危险工具。
- 需要更明确的用户确认。
- 第一版可以先实现但默认谨慎开放，也可以在前几个工具稳定后再开放给模型。

## 工具抽象

建议保留工具来源字段：

```ts
type AIToolSource = 'builtin' | 'custom' | 'mcp';
type AIToolPermission = 'read' | 'write' | 'dangerous';
```

工具定义描述模型可见的信息：

```ts
interface AIToolDefinition {
  name: string;
  description: string;
  source: AIToolSource;
  permission: AIToolPermission;
  parameters: unknown;
}
```

工具执行器描述应用内部如何执行：

```ts
interface AIToolExecutor<TInput = unknown, TResult = unknown> {
  definition: AIToolDefinition;
  execute: (input: TInput, context: AIToolContext) => Promise<AIToolExecutionResult<TResult>>;
}
```

工具上下文由当前编辑器页面注册到全局 Editor Context Store，工具执行器从 store 读取当前 active context：

```ts
interface AIToolContext {
  document: {
    id: string;
    title: string;
    path: string | null;
    getContent: () => string;
  };
  editor: {
    getSelection: () => EditorSelection | null;
    insertAtCursor: (content: string) => Promise<void>;
    replaceSelection: (content: string) => Promise<void>;
    replaceDocument: (content: string) => Promise<void>;
  };
}
```

实际实现时需要避免 `any`，`parameters` 和输入输出优先使用具体类型；无法确定时使用 `unknown` 并做运行时校验。

## 工具上下文注入

工具上下文采用全局 Editor Context Store 注入。

原因是当前布局中，`BChatSidebar` 位于默认布局里，编辑器页面位于 `router-view` 中，两者不是直接父子组件关系。如果通过 props 从 `views/editor/index.vue` 逐层传到 `BChatSidebar`，会让 layout 和 sidebar 绑定过多编辑器细节；如果使用 `provide/inject`，组件层级也不稳定。

推荐新增轻量上下文模块，例如：

- `src/stores/editor-context.ts`
- 或 `src/ai/tools/editor-context.ts`

当前编辑器页面 `src/views/editor/index.vue` 同时拥有 `fileState` 和 `editorRef`，因此由它在挂载时注册当前上下文，卸载时注销。

上下文中不要存静态 `content` 快照，而是存 `getContent()` 方法。聊天和工具调用可能持续数秒，用户在此期间可能继续编辑文档，工具执行时必须读取最新内容。

建议上下文结构：

```ts
interface EditorToolContext {
  document: {
    id: string;
    title: string;
    path: string | null;
    getContent: () => string;
  };
  editor: {
    getSelection: () => EditorSelection | null;
    insertAtCursor: (content: string) => Promise<void>;
    replaceSelection: (content: string) => Promise<void>;
    replaceDocument: (content: string) => Promise<void>;
  };
}
```

编辑器页面注册上下文时，大致形态如下：

```ts
const context = computed<EditorToolContext>(() => ({
  document: {
    id: fileState.value.id,
    title: fileState.value.name,
    path: fileState.value.path,
    getContent: () => fileState.value.content
  },
  editor: {
    getSelection: () => editorRef.value?.getSelection() ?? null,
    insertAtCursor: (content: string) => editorRef.value?.insertAtCursor(content) ?? Promise.resolve(),
    replaceSelection: (content: string) => editorRef.value?.replaceSelection(content) ?? Promise.resolve(),
    replaceDocument: async (content: string): Promise<void> => {
      fileState.value.content = content;
      editorRef.value?.setContent(content);
    }
  }
}));
```

`BChatSidebar` 或内置工具执行器不直接依赖编辑器组件，而是通过上下文 store 获取当前 active editor context：

```ts
const context = editorToolContextStore.getCurrentContext();
```

后续如果出现多个编辑器实例或多窗口场景，可以在 store 中增加 `activeEditorId`，由当前路由或 tabs 状态决定哪个上下文是 active context。

## 数据流

1. 用户在 `BChatSidebar` 中输入请求。
2. `BChat` 获取当前可用模型配置。
3. 工具注册表将内置工具转换为模型可识别的 tools schema。
4. `useChat` 发起带 tools 的流式请求。
5. Electron 主进程调用 `streamText`。
6. 主进程遍历 `fullStream`：
   - `text-delta` 继续发送给 renderer。
   - `finish` 继续发送 token usage。
   - `tool-call` 发送给 renderer。
   - `error` 发送给 renderer。
7. renderer 收到 tool call：
   - 只读工具自动执行。
   - 写工具展示确认弹窗。
   - 用户确认后执行工具。
   - 用户取消时返回取消结果。
8. renderer 将 tool result 加入下一轮模型消息，继续调用 stream。
9. 模型生成最终自然语言回复。
10. assistant 最终回复写入聊天历史。

## 用户确认

只读工具不打断用户。

写工具执行前必须显示确认信息：

- 工具名称。
- AI 请求执行的动作。
- 参数摘要。
- 对文档的影响范围。
- 对于替换类操作，尽量展示原文和新内容预览。

示例：

```text
AI 想要替换当前选区

原文：
...

新内容：
...

[取消] [应用]
```

取消不是错误，而是一个正常工具结果。模型应该收到“用户取消执行”的结果，再继续回复用户。

## UI 表现

聊天中可以展示工具调用状态，但不要把原始 JSON 暴露给普通用户。

建议状态：

- 正在读取当前文档。
- 正在搜索当前文档。
- 等待确认：替换选区。
- 已替换选区。
- 用户取消了工具调用。
- 工具执行失败。

开发期可以保留调试信息，但不作为默认用户界面。

## 错误处理

工具执行失败需要返回结构化结果，而不是直接中断整个聊天：

- 工具不存在。
- 参数校验失败。
- 当前没有打开文档。
- 当前没有选区。
- 用户取消。
- 编辑器不支持当前动作。
- provider 不支持 tools。

如果 provider 暂不支持 tools，可以后续考虑降级策略，但第一版优先实现原生 tool calling 链路。

  - 保持通用聊天组件边界，不写死具体编辑器工具。

  - 注入当前编辑器上下文和内置工具集合。

  - 新增工具类型、注册表、内置工具实现。

## 测试策略

优先覆盖工具协议和执行器：

- 工具注册表能正确返回可用工具。
- 只读工具能读取当前文档和选区。
- 写工具在未确认时不会执行。
- 用户取消写工具时返回取消结果。
- 无选区时 `replace_selection` 返回明确失败。
- 工具执行失败不会导致整个聊天状态卡死。

集成验证：

- 普通聊天仍然可用。
- 工具调用后 assistant 能继续生成最终回复。
- 流式文本、finish usage、abort 仍然正常。
- 历史会话只保存最终用户/assistant 消息，除非后续决定保存工具事件。

## 暂不做

- 用户自定义工具配置界面。
- MCP 工具发现和执行。
- Shell/文件系统等高权限工具。
- 工具调用历史的完整可视化时间线。
- provider 不支持 tools 时的 prompt JSON 降级。

这些能力后续可以在同一套工具抽象上继续扩展。
