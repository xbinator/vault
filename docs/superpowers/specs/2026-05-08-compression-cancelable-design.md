# 可取消压缩任务设计

## 背景

当前 `/compact` 手动压缩由 `useCompression` 独立执行，只维护自己的 `compressing` 状态。聊天发送链路中的 `loading` 和 `abort` 能力由 `submitUserTextMessage()` / `useChatStream` 管理，两者彼此分离。

这会带来两个问题：

- 压缩进行中无法复用现有发送态的“停止”交互
- 压缩任务被用户中止时，没有统一的任务收尾语义

本次设计目标是让手动压缩接入现有发送态框架，但不伪造一条用户文本消息。

## 目标

- `/compact` 执行时复用现有聊天发送态 `loading`
- `/compact` 执行时支持通过现有 `abort` 入口中止
- 中止后将当前压缩消息持久化更新为 `已取消`
- `已取消` 压缩消息不推进上下文边界

## 非目标

- 不实现“暂停后恢复继续”
- 不将压缩任务改造成真正的 LLM 流式消息
- 不新增假的 user message 作为 `/compact` 占位
- 不改变成功压缩消息作为上下文边界的既有规则

## 方案总览

采用“压缩任务接入发送态框架，但保持独立任务类型”的方案。

- `/compact` 仍先插入一条持久化 `compression` 消息，状态为 `pending`
- 压缩流程仍调用压缩协调器，不走 `agent.stream()`
- `useChatStream` 增加统一的“当前运行任务”概念，至少区分：
  - `chat`
  - `compression`
- 顶部与工具栏的 `loading` / `abort` 继续沿用 `useChatStream` 暴露接口
- 当当前任务是 `compression` 时，`abort` 不调用 `agent.abort()`，而是触发压缩取消信号

## 任务模型

建议在 `useChatStream` 内增加轻量任务控制状态：

- `activeTaskType: 'chat' | 'compression' | null`
- `compressionAbortController` 或等价取消信号

行为规则：

- 普通聊天发送开始时：`activeTaskType = 'chat'`
- 手动压缩开始时：`activeTaskType = 'compression'`
- 任一任务完成、失败或取消后：`activeTaskType = null`

`loading` 的定义保持“当前是否存在活跃任务”。

## 压缩取消语义

压缩任务取消采用 `AbortController` 风格语义：

- 开始压缩时创建取消信号
- 在压缩协调器的关键异步阶段检查是否已取消
- 若已取消，抛出受控取消错误
- 上层捕获后，将压缩消息状态更新为 `cancelled`

取消态文案：

- 主标题：`压缩已取消`
- 副标题：`此次上下文整理已停止，后续可重新发起压缩`

## 消息状态模型

当前压缩消息状态需要扩展为支持取消态。

建议在现有 `ChatCompressionStatus` 中新增：

- `cancelled`

状态语义：

- `pending`：正在压缩
- `success`：压缩完成，可作为边界
- `failed`：执行失败，不作为边界
- `cancelled`：用户主动中止，不作为边界

边界判断保持不变：只有 `success` 可成为模型上下文新边界。

## 执行流程

### `/compact` 触发

1. 插入持久化 `compression(pending)` 消息
2. 将全局运行任务切到 `compression`
3. 启动压缩协调器并传入取消信号

### 压缩成功

1. 更新同一条压缩消息为 `success`
2. 回填 `summaryId`、`coveredUntilMessageId`、`sourceMessageIds`
3. 清理活跃任务状态

### 压缩失败

1. 更新同一条压缩消息为 `failed`
2. 写入简短错误信息
3. 清理活跃任务状态

### 用户中止

1. 点击现有 `abort`
2. 若当前任务为 `compression`，触发取消信号
3. 压缩流程在最近的取消检查点停止
4. 更新同一条压缩消息为 `cancelled`
5. 清理活跃任务状态

## 与上下文边界的关系

取消态与失败态一致：

- 会保留在会话记录中，供用户看到本次任务被停止
- 不参与后续模型边界裁剪
- 不替代上一条成功压缩边界

因此再次执行 `/compact` 时：

- 若此前存在成功压缩边界，则从最后一条成功压缩消息之后开始
- 若此前从未成功压缩，则从整段会话开始

## 组件展示要求

`BubblePartCompression.vue` 需要支持取消态展示：

- 主标题：`压缩已取消`
- 副标题：`此次上下文整理已停止，后续可重新发起压缩`
- 视觉权重介于失败态与中性态之间，避免误导为系统错误

## 风险与约束

- 现有压缩链路中部分阶段为串行异步流程，取消只能在阶段边界生效，不能保证瞬时中断
- 若某一步已经完成数据库写入，再触发取消时必须保证最终状态一致，不产生“已保存摘要但消息显示已取消”的冲突
- 取消错误必须与真实失败区分，避免把用户主动中止显示成系统报错

## 验收标准

- `/compact` 执行时沿用现有发送态 `loading`
- UI 上可通过现有停止按钮中止压缩
- 中止后压缩消息显示 `压缩已取消`
- 取消态不会成为新的上下文边界
- 再次压缩时仍从最后一条成功压缩边界之后开始
