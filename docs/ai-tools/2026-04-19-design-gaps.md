# AI Tools 设计补充：缺口与待补项

日期：2026-04-19

## 背景

`docs/ai-tools/2026-04-18-design.md` 和 `docs/ai-tools/2026-04-18-implementation-plan.md` 已经确定了 AI Tools 的主方向：

- 第一版由 renderer 编排工具执行。
- Electron 主进程继续负责模型调用与流式事件转发。
- MVP 聚焦当前编辑器文档的读取与写入协作。
- 写工具必须经过用户确认。

这套方案本身是成立的，但如果直接开始实现，仍然会遇到一批没有完全定稿的边界问题。本文档用于补齐这些空白，帮助后续实现减少返工。

## 总体判断

当前缺口主要分为两类：

- 设计层缺口：协议、交互、安全边界还不够具体。
- 实施层缺口：状态机、测试、异常场景和分阶段交付还不够细。

建议把这些内容补充到后续实现过程中作为显式检查项，而不是默认开发时边做边猜。

## 设计层缺口

### 1. Tool Call 协议关联信息不完整

现有文档已经提出 `AIToolCall`、`AIToolResult` 等抽象，但还缺少足够稳定的关联字段。

建议补充：

- 每次工具调用都带唯一 `callId`。
- 工具结果必须显式引用对应 `callId`。
- 明确一次 assistant 回复内是否允许多个工具调用。
- 明确多个工具调用是串行还是允许并行。
- 明确 tool result 回注模型时的消息结构。

如果没有这部分定义，renderer 在处理连续工具调用和重试时容易出现结果错绑。

### 2. 参数校验策略不够明确

当前设计中 `parameters` 仍然比较抽象，缺少运行时校验约束。

建议补充：

- 工具输入统一使用可序列化对象。
- 每个工具定义独立的输入类型和运行时校验函数。
- 校验失败统一返回 `INVALID_INPUT`，并附稳定错误消息格式。
- 搜索关键词、替换内容、插入内容等字段都需要最小和最大长度约束。

类型声明只能约束开发期，renderer 与主进程之间仍然需要运行时防线。

### 3. 写工具确认策略还不够细

现有文档明确了“写操作必须确认”，但没有把确认策略具体化。

建议补充：

- `write` 与 `dangerous` 两类工具使用不同确认级别。
- `replace_document` 必须展示更强警示文案。
- 明确确认是否一律按“单次调用确认”，第一版不引入“本轮对话临时授权”。
- 用户关闭弹窗与主动点击取消都统一视为 `USER_CANCELLED`。

第一版应优先选择最保守的确认模型，避免后续再收紧权限时破坏交互习惯。

### 4. 写入预览形式未定

对于替换类工具，仅展示“工具名称 + 参数摘要”通常不够。

建议补充：

- `insert_at_cursor` 展示插入内容预览。
- `replace_selection` 展示原文和新内容的对照预览。
- `replace_document` 至少展示变更摘要，理想情况下展示 diff 或首尾片段预览。
- 当内容过长时需要截断，并提示已省略部分内容。

没有预览时，用户很难判断 AI 即将做出的真实修改。

### 5. 并发编辑场景缺少定义

工具调用期间，用户可能继续编辑文档，导致上下文发生变化。

建议补充：

- 写工具执行前重新读取一次当前文档和选区状态。
- 如果选区已变化，`replace_selection` 返回 `STALE_CONTEXT` 或 `NO_SELECTION`，而不是盲目覆盖。
- `replace_document` 在第一版中可以直接以“当前最新全文”为基准替换，但需要在确认文案里提醒它会覆盖整个文档。
- 后续如需增强，可引入版本号或内容哈希比对。

这是最容易在真实交互中出现的边界之一，值得提前约定。

### 6. 会话历史与工具事件的持久化策略未定

当前文档只提到“将工具结果加入下一轮模型消息”，但没有明确工具事件在本地会话中的保存方式。

建议补充：

- 第一版默认不把工具事件持久化为普通聊天消息。
- UI 中可以展示临时状态，但不进入长期历史记录。
- 如果未来需要调试能力，再单独设计“工具时间线”或开发者视图。
- 刷新页面或重进会话后，不恢复已完成的工具状态气泡。

这样可以先保证聊天历史简洁，避免一开始把产品交互做得过重。

### 7. Provider 能力矩阵还未明确

现有文档提到“优先实现原生 tool calling”，但还缺少 provider 侧的可用性定义。

建议补充：

- 列出当前接入 provider 中支持原生 tools 的范围。
- 不支持 tools 的 provider 在 UI 上禁用工具能力，或在发送前直接提示。
- 第一版不做 prompt JSON 降级，避免双轨逻辑。
- `UNSUPPORTED_PROVIDER` 应作为明确错误分支存在。

这部分越晚补，越容易在接入层堆条件分支。

### 8. 安全边界还可以更具体

虽然第一版工具范围较小，但仍然需要基本限流和截断策略。

建议补充：

- 限制单次读取全文返回给模型的最大长度。
- 搜索结果的匹配条数与上下文片段长度设上限。
- 单次插入和替换内容长度设置上限。
- 超长内容在返回时增加 `truncated` 标记或明确说明。

这不仅影响安全，也影响模型上下文成本和 UI 可读性。

## 实施层缺口

### 1. 工具注册与模型 Schema 转换职责不够清晰

文档中已经提到“将内置工具转换为模型可识别的 tools schema”，但还缺少显式职责模块。

建议补充：

- 新增注册表或转换层，专门负责把内部 `AIToolDefinition` 转成 provider 所需 schema。
- `BChat` 只消费可用工具列表，不负责拼接具体 schema。
- provider 特定字段映射收敛在接近 AI service 的位置。

这能避免后续 provider 差异污染业务层组件。

### 2. 中止与超时场景未定义

当前数据流没有清晰说明 abort 行为对工具状态的影响。

建议补充：

- 用户点击停止后，不再接受新的工具调用。
- 已进入确认态但尚未操作的工具，统一关闭并标记为已取消本轮流程。
- 工具执行过程如果超时，返回 `TOOL_TIMEOUT` 或 `EXECUTION_FAILED`。
- 模型流被中止后，renderer 不再继续发起下一轮工具结果请求。

如果这里没有状态定义，很容易在“停止生成”后留下悬空确认框或脏状态。

### 3. 需要幂等与重入保护

流式事件和 UI 交互都存在重复触发的可能。

建议补充：

- 同一个 `callId` 只允许提交一次工具结果。
- 相同工具事件重复到达时直接忽略或覆盖为最新状态，但不能重复执行写操作。
- 在等待工具确认时，发送按钮和再次触发同一流程的入口需要有保护。

对写工具来说，幂等保护是必要条件，不是锦上添花。

### 4. 错误码集合还可以扩展

当前错误码已经覆盖基础情况，但不够完整。

建议补充：

- `NO_CURSOR`
- `STALE_CONTEXT`
- `TOOL_TIMEOUT`
- `UNSUPPORTED_PROVIDER`
- `CONFIRMATION_DISMISSED`

是否全部在第一版暴露给模型可以再斟酌，但内部实现最好先预留。

### 5. UI 状态机还没有明确描述

当前只是列出了几种状态文案，缺少状态转换定义。

建议补充以下最小状态：

- `idle`
- `streaming_text`
- `executing_tool`
- `awaiting_tool_confirmation`
- `resuming_after_tool`
- `completed`
- `failed`
- `aborted`

同时明确：

- 何时允许用户继续输入。
- 工具确认时是否阻塞输入框。
- 取消工具后是否继续保留本轮 assistant 草稿状态。

### 6. 双编辑器实现差异需要提前对齐

计划中同时修改富文本编辑器与源码编辑器，但二者的选区与替换语义未必完全一致。

建议补充：

- `BEditorPublicInstance` 对外暴露统一语义，不泄漏内部实现差异。
- 富文本模式下的 `getSelection()` 返回结果必须保证可序列化、可解释。
- 两种编辑模式对“插入到光标”“替换选区”“替换全文”的失败条件要统一。

不先统一边界，后面测试会出现模式相关分叉。

### 7. 集成测试场景偏少

现有测试计划更偏单元测试，缺少关键链路集成验证。

建议至少补充以下场景：

- 只读工具自动执行后，assistant 能继续生成最终回复。
- 写工具确认后，修改成功并继续生成最终回复。
- 写工具取消后，模型收到取消结果并改为自然语言说明。
- provider 不支持 tools 时，聊天不会卡死，并给出明确提示。
- 用户中止流式回复后，工具链路不会继续推进。

### 8. 需要明确分阶段交付顺序

实现计划很完整，但第一版实际落地最好再拆出更小的可交付切片。

建议交付顺序：

1. 打通 tool call 事件透传链路，但只接只读工具。
2. 实现 `read_current_document` 与 `get_current_selection`。
3. 接入 `search_current_document`，补齐只读测试。
4. 加入写工具确认框和 `insert_at_cursor`。
5. 再接 `replace_selection`。
6. 最后视稳定性决定是否开放 `replace_document`。

这样可以先验证“协议 + 流程 + UI 状态机”是否成立，再逐步增加风险更高的写能力。

## 建议新增的明确约束

为避免实现过程中的反复讨论，建议将以下约束直接写入主设计或实施计划：

- 第一版只支持单文档、单 active editor context。
- 第一版不支持工具并行执行。
- 第一版不做 provider 降级。
- 第一版不持久化工具事件历史。
- 第一版所有写工具都按单次调用确认。
- 第一版默认不向模型开放 `replace_document`，直到前序工具稳定。

这些约束并不意味着后续不能扩展，只是帮助第一版更快收敛。

## 建议的后续动作

如果要继续完善 `docs/ai-tools`，建议接下来按顺序补三件事：

1. 在设计文档中补充协议字段、状态机和确认策略。
2. 在 implementation plan 中加入集成测试与分阶段交付说明。
3. 在真正开工前确认 `replace_document` 是否进入 MVP。

## 结论

当前 AI Tools 方案的主方向已经足够清晰，真正欠缺的不是“大方向”，而是“边界条件”和“落地顺序”。

如果这些补充项能在实现前写清楚，后续开发会更稳，聊天组件、编辑器组件和 Electron 链路之间的职责也会更容易守住。
