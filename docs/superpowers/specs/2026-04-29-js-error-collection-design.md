# JavaScript 错误收集系统 — 设计文档

## 概述

基于现有日志收集系统，扩展全局 JavaScript 错误收集能力。覆盖渲染进程、Preload 脚本和主进程的运行时错误，将各类未捕获异常统一收集到日志系统，提升应用稳定性监控和问题排查能力。

---

## 需求汇总

| 维度 | 决策 |
|------|------|
| 收集范围 | 全局 JS 运行时错误、未处理 Promise 错误、Vue 渲染错误、主进程未捕获异常 |
| 覆盖进程 | 主进程 + Preload + 渲染进程 |
| 错误级别 | 统一为 ERROR 级别 |
| 日志格式 | 复用现有日志系统格式，附加错误上下文信息 |
| 查看方式 | 复用设置页日志查看器，按 ERROR 级别筛选 |
| 架构模式 | 分层收集 → 统一上报 → 集中写入 |

**架构说明**：
- Preload 脚本**直接**通过 IPC 上报到主进程，**不经过**渲染层，避免循环依赖
- 脱敏处理统一在**主进程写入前**完成，确保所有入口（渲染层、Preload、主进程自身）的日志都经过脱敏

## 错误收集覆盖范围

### 1. 渲染进程错误类型

| 错误来源 | 收集方式 | 说明 |
|---------|---------|------|
| 全局 JS 运行时错误 | `window.onerror` | 同步/异步脚本执行错误 |
| 未处理的 Promise 错误 | `window.onunhandledrejection` | async/await 未捕获异常 |
| Vue 组件渲染错误 | `app.config.errorHandler` | 模板渲染、生命周期错误 |
| Vue 警告 | `app.config.warnHandler` | **仅开发环境**收集，用于调试 |

### 2. Preload 进程错误类型

| 错误来源 | 收集方式 | 说明 |
|---------|---------|------|
| 全局 JS 运行时错误 | `window.onerror` | preload 脚本自身错误 |
| 未处理的 Promise 错误 | `window.onunhandledrejection` | IPC 回调异常等 |

### 3. 主进程错误类型

| 错误来源 | 收集方式 | 说明 |
|---------|---------|------|
| 未捕获异常 | `process.on('uncaughtException')` | 已由 electron-log 覆盖 |
| 未处理 Promise | `process.on('unhandledRejection')` | 异步操作异常 |
| 进程警告 | `process.on('warning')` | Node.js 弃用警告等 |

## 设计原则

### 分层收集，统一上报

- **渲染进程**：通过 `error-collector.ts` 统一拦截各类错误，格式化后调用 `logger.error()` 上报
- **Preload 脚本**：直接调用 `writeScopedLog()` 写入，**不经过渲染层**，避免循环依赖
- **主进程**：扩展现有 `electron-log` 的错误捕获，补充 Promise 和 Warning 收集

### 错误信息标准化

每条错误日志包含以下字段：

```
[2025-04-29 10:23:45.123] [ERROR] [renderer] Error: TypeError: Cannot read properties of undefined
Stack: TypeError: Cannot read properties of undefined (reading 'name')
    at BEditor.vue:156:23
    at callWithErrorHandling (runtime-core.esm-bundler.js:155:22)
Context: {"component":"BEditor","info":"render function","url":"/editor/123"}
```

| 字段 | 说明 |
|------|------|
| Error Type | 错误类型名称 |
| Message | 错误消息 |
| Stack | 完整堆栈信息 |
| Context | JSON 格式的上下文（组件名、URL、额外信息） |

### 静默处理，不中断流程

- 错误收集器本身**永不抛出异常**
- 上报失败时静默处理，避免影响业务逻辑
- `window.onerror` 返回 `false`，**不阻止**浏览器默认的控制台输出（`return true` 才会阻止）

### 与现有日志系统复用

- 复用 `LogLevel.ERROR` 级别，日志查看器无需修改
- 复用 `LogScope` 标识区分错误来源
- 复用文件滚动、清理机制

## 错误信息脱敏

**脱敏时机**：主进程写入前统一脱敏

**原因**：
- 渲染层脱敏逻辑可能被绕过（Preload 直接调用 `writeScopedLog`，完全跳过了渲染层）
- 集中脱敏逻辑，便于统一维护和更新规则
- 防止不同进程实现不一致导致的安全漏洞

**脱敏规则**：

| 敏感类型 | 处理方式 | 示例 |
|---------|---------|------|
| API Key | 替换为 `[REDACTED_API_KEY]` | `sk-abc123...` → `[REDACTED_API_KEY]` |
| Access Token | 替换为 `[REDACTED_TOKEN]` | Bearer token 等 |
| 用户文件路径 | 保留文件名，脱敏完整路径 | `/Users/name/doc.md` → `.../doc.md` |
| 密码 | 完全移除 | 任何 `password` 字段 |

## 性能边界

### 错误上报频率限制

**作用域**：全局跨错误类型限制

> 设计考量：如果按错误类型独立计数，高频的某一类错误（如轮询失败）可能独占全部配额，导致其他更严重的错误无法上报。全局限制的作用是**避免单一错误类型独占带宽**，即使高频错误爆发，系统仍有机会记录其他类型的关键错误。

| 场景 | 限制策略 | 说明 |
|------|---------|------|
| 高频循环错误 | 10秒内最多上报10条 | 防止错误风暴导致日志膨胀 |
| 重复相同错误 | 5分钟内去重 | 基于错误消息哈希 |
| 大量堆栈 | 截断至 50 行 | 防止单条日志过大 |

### 去重缓存生命周期

- **存储位置**：内存（模块级变量）
- **生命周期**：当前进程生命周期内有效
- **进程重启**：缓存清除，同一错误会重复上报
- **设计意图**：
  - 进程崩溃后新实例启动，重新记录同类错误是合理的（可能是新问题）
  - 内存缓存轻量，无需持久化复杂度
  - 日志文件可通过外部工具做长期去重分析

### 内存占用

- 错误收集器自身状态：< 1KB
- 去重缓存（最近100条）：< 50KB
- 对整体应用性能影响可忽略

## 错误去重哈希策略

**当前方案**：`message` 前 100 字符简单哈希 + 基础规范化处理

**规范化规则（v1.0 实现）**：
- 连续数字序列（4位以上）替换为 `<NUM>`
- UUID 格式替换为 `<UUID>`
- 16进制哈希值（8位以上）替换为 `<HASH>`
- 示例：`Failed to load user_12345` 和 `Failed to load user_67890` 都规范化为 `Failed to load user_<NUM>`

**优化建议（v1.1 迭代）**：
- 组合哈希：`message + stack 前 3 行`
- 更智能的上下文相关规范化

## 主进程 uncaughtException 处理策略

**当前现状**：`electron-log` 已覆盖 `uncaughtException`

**风险与决策**：
- Node.js 官方建议收到 `uncaughtException` 后应**让进程退出**，因为进程状态可能已经不可信
- Electron 应用在主进程崩溃时会自动重启渲染进程，用户体验较差
- **实现前需确认**：`electron-log` 注册 handler 后是否会 fallthrough 到 Node.js 默认行为（打印堆栈并退出）。如果 `electron-log` 吞掉异常而不退出，需评估风险。
- **本系统策略**：
  1. 记录错误到日志（确保崩溃原因可追溯）
  2. **不主动调用 `process.exit()`**，由 Electron 默认行为决定
  3. 监控发现崩溃率异常时，再考虑强制退出策略

## Vue warnHandler 策略

| 环境 | 策略 | 原因 |
|------|------|------|
| 开发环境 | **启用**收集 | 便于调试 Vue 警告（如 prop 类型不匹配、已弃用 API） |
| 生产环境 | **禁用**收集 | Vue warning 量可能很大，写入日志文件的收益有限，避免日志噪音 |
