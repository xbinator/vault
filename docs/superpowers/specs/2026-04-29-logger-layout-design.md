# 日志查看器布局改版设计

## 概述

将设置页中的日志查看器从“普通列表 + 顶部筛选栏”调整为更贴近设计示例的时间轴卡片布局。

本次改版聚焦于前端展示层，不修改日志读取、筛选、分页和目录打开等业务逻辑。目标是在保留现有数据流和交互语义的前提下，提升日志列表的信息层级、可读性和整体一致性。

## 目标

- 将日志列表改为“左侧信息列 + 中轴圆点连线 + 右侧消息卡片”的结构。
- 让日志条目的视觉表现接近示例图，包括圆点颜色、卡片边框、圆角、留白和文字层级。
- 同步收敛页面容器和顶部过滤栏样式，使其与日志卡片属于同一套视觉语言。
- 保持现有功能不变，包括首次加载、筛选、关键字搜索、日期选择、触底加载更多、刷新和打开日志目录。
- 在不重构 store 的前提下完成改造，降低回归风险。

## 非目标

- 不新增日志详情抽屉、展开面板或复制功能。
- 不调整日志筛选字段和查询参数结构。
- 不修改 `logger.getLogs()` 的返回格式。
- 不引入虚拟列表或新的分页策略。
- 不新增悬停动画、长文本折叠等超出示例图范围的交互增强。

## 现状问题

- 当前日志项左右信息分离度不够，时间、级别、来源和正文的视觉权重接近，重点不够突出。
- 时间轴结构较弱，左侧更像简单列布局，缺少示例图中的“中轴”识别感。
- 顶部标题区、筛选区和内容区虽然功能明确，但背景、边框和留白层级较平，整体观感偏散。
- 日志正文直接铺在页面内容区中，缺少独立卡片承载，列表可扫描性一般。

## 总体方案

采用“页面统一改造”方案：

- 保留现有目录结构和组件拆分方式：`index.vue`、`LogFilterBar.vue`、`LogTimeline.vue`、`logViewer.ts`。
- 将核心改造集中在模板结构和样式层。
- `LogTimeline.vue` 负责新的时间轴卡片布局。
- `LogFilterBar.vue` 负责顶部标题区和筛选区的视觉统一。
- `index.vue` 负责页面背景、内容容器和空态/加载态在新视觉中的落位。

这样可以最大程度复用现有数据流，同时把改动边界控制在页面展示层。

## 布局设计

### 页面结构

页面维持三段式结构：

1. 标题与操作区
2. 过滤区
3. 日志内容滚动区

视觉上不再强调整个页面是一个纯白大容器，而是通过更轻的背景色、统一的圆角和细边框，让顶部区域和日志卡片形成连续的界面语言。

### 日志项结构

每条日志使用三列布局：

1. 左侧信息列
2. 中轴列
3. 右侧内容列

左侧信息列展示：

- 时间：使用较高权重、等宽数字风格，便于纵向扫描。
- 来源：显示日志来源中文标签，例如“主进程”“渲染进程”“预加载脚本”。

中轴列展示：

- 一个按日志级别着色的圆点。
- 一条垂直延伸的连接线。

右侧内容列展示：

- 顶部级别标签，如“错误”“警告”“信息”。
- 下方消息正文，使用卡片承载。

该结构与示例图一致，能够让用户先扫时间，再看来源和级别，最后阅读正文。

### 对齐策略

- 左侧信息列采用固定宽度，避免因来源名称长度变化导致时间轴晃动。
- 中轴列宽度保持收敛，只承担圆点和连线，不承载文本。
- 右侧内容列自适应填充剩余空间。
- 日志项垂直间距统一，卡片顶部和圆点中心对齐。

## 视觉规则

### 页面与容器

- 外层页面背景调整为更轻的浅色背景感。
- 标题区和过滤区保留白底，但通过边框和间距与正文形成清晰分层。
- 内容滚动区保持足够的左右内边距，避免卡片贴边。

### 时间轴

- 中轴线使用浅灰色细线。
- 圆点维持按级别着色：
  - `ERROR` 使用红色
  - `WARN` 使用橙黄色
  - `INFO` 使用蓝色
- 圆点尺寸略大于当前实现，以增强时间轴存在感。

### 消息卡片

- 使用白底、浅边框和中等圆角。
- 卡片内部使用“标签在前、消息在后”的横向或轻量纵向组合，优先贴近示例图的单行阅读体验。
- 正文在长文本场景下继续允许换行，避免内容被截断。
- 正文使用较高对比度的深色文本，来源和时间使用次级文本色。

### 顶部工具区

- 标题“运行日志”和记录数保留，但记录数字段的视觉权重降低，避免与标题竞争。
- 操作按钮继续保留“刷新”“打开目录”，统一为更轻的文本按钮风格。
- 过滤控件统一高度、圆角和间距，避免当前工具条中不同控件的节奏不一致。

## 组件边界

### `src/views/settings/logger/index.vue`

负责：

- 页面整体容器结构
- 空态和加载态在新布局中的位置
- 滚动容器和触底加载

需要调整：

- 页面背景与内容区内边距
- 空态和加载态与新卡片式布局的视觉协调

不需要调整：

- `handleScroll()` 的业务逻辑
- 初次加载时机

### `src/views/settings/logger/components/LogFilterBar.vue`

负责：

- 标题区与操作区
- 过滤栏控件布局

需要调整：

- 标题区和过滤区的分层样式
- 按钮、筛选控件和间距的统一
- 记录数文案的弱化展示

不需要调整：

- `computed` 双向绑定方式
- `store` 调用方式

### `src/views/settings/logger/components/LogTimeline.vue`

负责：

- 单条日志的结构编排
- 时间轴和消息卡片视觉呈现

需要调整：

- 模板从当前两块式布局升级为三列式时间轴结构
- 来源信息从右侧头部移到左侧信息列
- 级别标签与正文放入独立卡片容器
- 时间轴圆点、连线、卡片间距和对齐关系

不需要调整：

- `entries` 输入数据结构
- `getLogLevelColor()`、`getLogLevelLabel()`、`getLogScopeLabel()` 的职责

## 数据与行为

本次改版不改变以下逻辑：

- 页面挂载时调用 `store.loadLogs(true)`。
- 修改任一筛选条件时，仍由 store 直接触发重新加载。
- 内容滚动到底部时，仍由 `store.loadMore()` 执行增量分页。
- 空列表时显示空态。
- 加载中时显示加载指示器。

也就是说，本次改造是纯展示层重构，不引入新的业务状态。

## 风险与处理

### 风险 1：布局调整影响长文本换行

日志正文可能包含较长英文、URL 或堆栈内容。若卡片结构和 `flex` 配置不当，可能导致文本溢出或挤压左侧列宽。

处理方式：

- 保持右侧内容列 `min-width: 0`。
- 保留 `word-break: break-all` 和 `white-space: pre-wrap`。

### 风险 2：左侧列宽不稳定影响整体整齐度

来源标签长度不一，若左列宽度不固定，时间轴会左右抖动。

处理方式：

- 左列采用固定宽度。
- 来源文本居中或右对齐，以维持视觉稳定。

### 风险 3：顶部区域和正文区风格割裂

如果只改日志项不改顶部工具区，页面会出现明显风格分裂。

处理方式：

- 同步调整 `LogFilterBar.vue` 和 `index.vue` 的容器层级、边框和留白。

## 验收标准

- 日志列表整体结构与示例图一致：左侧时间和来源、中轴圆点连线、右侧消息卡片。
- 日志级别颜色映射保持正确，且不同级别在视觉上可快速区分。
- 顶部标题区和过滤区与正文卡片风格统一，不再显得割裂。
- 原有筛选、刷新、打开目录和触底加载功能行为保持不变。
- 长文本日志能够正常换行展示，不出现横向溢出。
- 空态和加载态在新布局中仍然清晰可用。

## 测试与验证

本次以手工验证为主，不新增自动化测试。

验证重点：

- 不同级别日志的颜色、标签和圆点样式是否正确。
- 来源字段是否正确出现在左侧信息列。
- 长消息、短消息和空列表场景的显示是否正常。
- 滚动加载更多后，新增日志项的对齐和连线是否正常。
- 顶部过滤栏在不同窗口宽度下是否仍能正常排布。

## 实施边界

本次实现仅覆盖现有页面视觉改版，不额外引入如下内容：

- 日志详情弹窗
- 展开/收起长消息
- 复制日志内容
- 更复杂的 hover 态和交互动效

后续若需要进一步增强体验，应单独出设计文档，不在本次改版中一并实现。
